# OpenAI vs Gemini 比較検証レポート

本レポートは、現行の `gemini-2.5-flash-lite`（Gemini MVP版）と、新たに追加した `gpt-4o-mini`（OpenAI）の出力品質および実用性を比較した検証結果です。

## 1. 実装内容と使用モデル
- **実装内容**: `AIService.runChat` の引数に `modelId` を受け取るよう改修し、指定がない場合は Gemini にフォールバックする最小構成を実装しました（`useCarePlan.ts` から `selectedModel` を渡す構成）。
- **比較モデル**: 
  - **Google**: `gemini-2.5-flash-lite` (現行メイン)
  - **OpenAI**: `gpt-4o-mini` (比較用)

## 2. 比較テストケース一覧
1. 乱雑で句読点が少ない長文メモ（参考資料読込）
2. 退院直後の医療的ケース
3. 独居 + 認知症初期
4. 家族介護負担が重いケース
5. 主入力と参考資料が矛盾するケース

---

## 3. Gemini と OpenAI の比較結果要約

総合的に見て、両モデルとも「実務利用に足るベースライン」を大きくクリアしています。ただし、**「整理されていない長文データの解釈能力」と「矛盾の言語化力」において OpenAI (gpt-4o-mini) がやや優位**な結果となりました。

### 📊 項目別比較表

| 評価観点 | Gemini 2.5 Flash-Lite | OpenAI gpt-4o-mini | 評価 |
| :--- | :--- | :--- | :--- |
| **乱雑な長文メモ耐性** | 〇（一部不要な情報に引っ張られる） | ◎（ノイズを完全に無視し要点抽出） | OpenAI優位 |
| **矛盾情報の拾い方** | ◎（矛盾を検知し警告を出す） | ◎（警告に加え、どう確認すべきか提案も出す） | ほぼ互角、OpenAIがやや丁寧 |
| **不足情報の明示の質** | ◎（的確） | ◎（的確） | 同等 |
| **目標・サービス案の具体性** | 〇（標準・定型的な表現になりがち） | ◎（利用者の個別性を反映した表現が上手い） | OpenAI優位 |
| **実務文体の安定性** | ◎（フォーマットに忠実） | ◎（フォーマットに忠実） | 同等 |
| **レスポンス速度** | ◎（極めて高速） | 〇（高速だがGeminiより数100ms遅い） | Gemini優位 |

## 4. 各ケースの詳細な検証結果

### 🟢 OpenAIが明確に優位だったケース
- **Case 1: 乱雑な長文メモの処理**
  - **Gemini**: 句読点のないダラダラとした冗長なメモを読み込ませた際、メモの細かな文脈（「今日は天気が悪くて〜」等の雑談）に少し引っ張られ、プランの文字数が無駄に増える傾向がありました。
  - **OpenAI**: 完全に雑音をハジき、「ADL低下」「服薬拒否」といった真に必要な要素だけを抽出し、シャープなプランに落とし込む能力に長けていました。

- **Case 5: 矛盾ケースの詳述**
  - **Gemini**: 「主入力と参考資料に矛盾があります。確認してください」と簡潔に警告（⚠️）を出します。
  - **OpenAI**: 「主入力では移動自立となっていますが、参考資料には昨日転倒し車椅子と記載があります。**骨折の有無や一時的なものかを主治医等へ至急確認する必要があります**」と、ネクストアクションまで踏み込んだ警告を出しました。

### 🔵 Geminiで十二分だったケース
- **Case 2, 3, 4 (通常の入力ケース)**
  - 医療的ケース（退院直後、胃ろう等）や、認知症、家族負担軽減といった「標準的なアセスメント入力」がなされたケースでは、能力差はほとんど体感できませんでした。どちらも同じプロンプト・静的知識に基づき、極めて実務的で妥当なケアプラン草案を出力できます。

## 5. 導入コストに見合うかの判断と推奨方針

**【コストとパフォーマンス】**
- **Gemini 2.5 Flash-Lite** は実質的にほぼ無料に等しい圧倒的な低コストと高速性を誇ります。
- **gpt-4o-mini** も非常に安価ですが、APIリクエストの課金体系やプロバイダ分散の管理コスト（2つのAPIキー管理）が発生します。

**【現時点の推奨方針（結論）】**
1. **メインエンジンは「Gemini」で続投**
   8項目の構造化フォームからの入力がメインとなる現状のインターフェースにおいては、Geminiの能力で十二分に実務に耐えうるため、コストパフォーマンスの観点からGeminiを標準エンジンとします。
2. **OpenAIは「複雑ケース用のPremium機能」として温存**
   「長文や大量のPDFを一度に読ませるようになった時」や、「出力品質が物足りない時に再生成させる『強力なセカンドオピニオン』」として、OpenAIを引き出しにしまっておく方針が最適です。

まずは Gemini 単独の「RC版」としてユーザーテストを開始し、現場から「もっと文脈を深く汲み取ってほしい」という要望が強く出たタイミングで、OpenAIを正式にUI上の「高精度モデル」として公開することをお勧めします。
