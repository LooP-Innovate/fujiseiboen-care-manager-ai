## 2. 検証による評価

### ✅ うまくいった点（強み）
- **課題抽出の妥当性**: 「本人の希望」と「家族の懸念」にズレがある場合（Case 2など）、その両方を汲み取ったバランスの良い課題抽出ができている。
- **実務に直結する目標設定**: Case 3のように、本人の希望（入浴）に直結した現実的な短期・長期目標が設定される。
- **不足情報の明示が強力**: 例えばCase 1では「家の構造（持ち家か賃貸か）」「退院時のリハビリサマリーの有無（Case 3）」など、ケアマネジャーが追加でアセスメントすべき事柄が的確に『不足情報・注意点』としてリストアップされ、実務上の抜け漏れ防止に役立つ。

### ⚠️ 弱かった点・限界
- **支給限度額の概念の欠如**: 要介護1（Case 1）に対して、連日の訪問介護などを無邪気に提案してしまうリスクがある。（AIは点数計算までは考慮しないため）
- **医療的深度の不足**: 重度医療（Case 5）において、「たん吸引の具体的な頻度・主治医による指示書の詳細」などを推測で書ききれないため、サービス案がやや一般論（「訪問看護の導入」等）に留まる。
- **フォーマル/インフォーマル資源の区別**: 地域の独自サービスやボランティア等の「インフォーマルな支援」の提案が弱く、介護保険サービス（フォーマル）に偏りがち。

## 3. 入力項目の追加・修正要否

現行の**「8項目への最適化フォーム」で十分に高品質なドラフトが生成できる**ことが確認されました。むやみに入力欄を増やすとケアマネの手間が増えるため、**Phase 1での入力フォームはこれで完成**とします。

ただし、将来的（Phase 2以降）には以下の対応が求められます：
- 保険外サービスやローカルな支援状況を入力できる「地域の独自資源メモ」の追加。
- 支給限度額アラート（UI側での警告）。

---

## 4. Phase 2 の優先強化候補（ネクストアクション）

今回の検証結果から、Phase 2（高度化）では以下の機能開発を優先すべきと考えられます。

1. **RAG（検索拡張生成）によるローカル知識の付与**
   - 「この地域で利用できる医療的ケア対応のショートステイ」や「インフォーマルサービス一覧」などの独自ドキュメントを読み込ませ、より地域に根ざした具体的なサービス提案を可能にする。
2. **長文ファイル（PDF / テキスト）の直接読み込み**
   - 病院からの「退院支援サマリー」や、認定調査の「特記事項」等をそのままアップロードして解析できるようにし、フォーム手入力の手間をさらにゼロに近づける。
3. **OpenAI（GPT-4o系）等へのモデル切り替え・比較機能**
   - 複雑な医療的ケースにおける推論力や、文章の微調整力がGeminiとどう違うか、モデルを切り替えてテストできる基盤を提供する。
